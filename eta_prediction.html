Train:

  • city_id - City in which task is created
  • drop_area - Area from which user raised a task
  • store_dzid - dunzo store id
  • pickupareaid - Represents the Area from where dunzo partner needs to pickup
  • store_name - name of the store. Two stores with different dzid can have the same names because they probably belong to the same food chain
  • sub_tag - Kind of task Ex:- Food, Grocery, Pets
  • task_created_on - Timestamp at task creation time
  • accepted_time - Timestamp at dunzo delivery partner acceptance (NOT IN TEST)
  • reached_for_pickup_time - Timestamp of dunzo partner reaching the store (NOT IN TEST)
  • purchased_time - Timestamp at which partner successfully made a purchase of ordered items (NOT IN TEST)
  • start_for_delivery_time - Timestamp at which partner started for delivery to the user from store or pickup location. (NOT IN TEST)
  • reached_for_delivery_time - Timestamp at which partner reached the drop point or user address (NOT IN TEST)
  • delivered_time - Timestamp at which partner successfully delivered the items (NOT IN TEST)
  • rt_estimated_pickup_distance - estimated distance for partner to reach pickup point or store
  • rt_estimated_task_distance - estimated distance from pickup point to drop point
  • Target - subtract delivered_time timestamp from task_created_on timestamp

Any delivery partner related temporal attributes are not present in the test set. The task is to predict the ETA which is time taken from the point when the task was created to when it could get completed.

In order to calculate ETA, we need a complete fill rate on all temporal attributes so we need to discard observations that have these values missing. Why is this necessary? We could have just taken a difference between delivered_time and task_created_on, however, the assumption then would be that all tasks created get fulfilled in less than an hour. A more reasonable assumption is that the difference in times between each step in the task completion process is less than an hour. These steps are:

Task Created -> Accepted -> Reached For Pickup -> Purchased -> Start For Delivery -> Reached For Delivery -> Delivered

What we'll do below is calculate difference between each of these steps (successively) and then add up the times to create the target label. This process would have been a lot easier, if we had the day + hour in the timestamp

def convert_timestamp(col):
    # Getting rid of the decimal point after seconds
    df_train[col] = df_train[col].apply(lambda x: x.split('.')[0])
    return(pd.to_datetime(df_train[col], format='%M:%S'))

# A very small number of values is missing in pickupareaid so we'll remove them as well
df_train = df_train.dropna(subset=temporal_cols + ['pickupareaid'])
for col in temporal_cols:
    df_train[col] = convert_timestamp(col)
for col_index in range(len(temporal_cols)-1):
    df_train['diff_'+str(col_index)] = df_train[temporal_cols[col_index+1]] - df_train[temporal_cols[col_index]]
    # To get target value in seconds. The if condition just takes care of those observations where time difference is -ve
    df_train['diff_'+str(col_index)] = df_train['diff_'+str(col_index)].apply(lambda x: (3600+x.total_seconds())
                                                                                        if x.days==-1 else x.total_seconds())
df_train['time_taken'] = df_train[[col for col in df_train.columns if 'diff' in col]].sum(axis=1)

# The temporal attributes in themselves have no predictive power since we don't have this data at an hour/day level
# where one would expect task completion times to be greater during peak hours and weekdays

For the other attributes, this is a very small number wrt the cardinality of the variable. The levels that are present in test but not in train are levels that have very low frequency. Such levels are also present in train but not in test. Since a small number of levels are responsible for majority of the observations, we can combine the low frequency levels under one bucket. These levels will be sorted from least to most frequent and the least frequent levels that have a cumulative frequency less than some threshold will be labeled as -1 or 'Other'. This has been discussed here (https://www.kaggle.com/c/bluebook-for-bulldozers/discussion/4066 and https://datascience.stackexchange.com/questions/37480/how-can-i-do-classification-with-categorical-data-which-is-not-fixed/37486)

# Low frequency columns are labeled as -1
def high_freq_enc(col, thresh=0.9):
    high_freq_level = df_data[col].value_counts(normalize=True).cumsum() < thresh
    high_freq_level = high_freq_level[high_freq_level==True].index
    return(df_data[col].apply(lambda x: x if x in high_freq_level else -1))

# We perform this labeling/encoding on the combined data. This is to make sure low frequency columns in either data sets -
# test and train are labeled as -1.
col_thresh = {'drop_area':0.98, 'pickupareaid':0.98, 'store_dzid':0.9, 'store_name':0.9}
for col in ['drop_area', 'pickupareaid', 'store_dzid', 'store_name']:
    df_data[col] = high_freq_enc(col, col_thresh[col])
    
# Seperating train and test
df_train = df_data.iloc[:train_last_row]
df_test = df_data.iloc[train_last_row:]

# The above method takes care of low frequency levels in drop_area and pickupareaid.
# About 4 levels in store_dzid and 1 level in store_name still remain. These will now be replaced with -1
def replace_values(col):
    values_missing = list(set(df_test[col]) - set(df_train[col]))
    df_test[col] = df_test[col].replace(values_missing, -1)

# To replace values in other attributes    
replace_values('store_dzid')
replace_values('store_name')

Feature Engineering¶

Mean encodings/ target encodings are values that are aggregates of target grouped by levels within categorical attribute(s). Since we do not have target values for the test set, what we can do is calculate them for the train set, store them in a dictionary and use them to map categorical attributes within the test set. In order to reduce overfitting, we will use a concept known as expanding mean. Expanding mean for a level within a categorical attribute considers cumulative sum of target value for all observations prior to the one it is being computed on (that belong to the level) and divides them by their cumulative count. Mean encodings help us create a lot of features and reduce the cardinality of the categorical variables. Some of the important questions that will help us create these mean encodings are listed below:

  • What is the average target value vs drop_area?
  • What is the average target value vs pickupareaid?
  • What is the average target value for the combination of drop_area, pickupareaid?
  • What is the average target value vs store_dzid?
  • What is the average target value vs store_name?

Do some areas have more delivery partners than others? When a user raises a task, how much time do delivery partners closest to the pickupareaid take to accept the task. This can be answered by computing another feature:

  • What is the average acceptance time vs pickupareaid?

Since we have also calculated time it takes for a partner to purchase items at a particular store, they can then be used to calculate more mean encodings. It is possible that some store chains are more efficient than others:

  • What is the average purchase time vs store_dzid?
  • What is the average purchase time vs store_name?

We have also computed a feature called speed (distance/time) for pickup and drop. We have been given estimated pickup and drop distance and by using time taken for the particular drop_area and pickupareaid, we can compute:

  • What is the average speed vs pickupareaid?
  • What is the average speed vs drop_area?

city_id and sub_task won't be used for creating mean encodings since there isn't a great difference in the average target variable for each city_id/sub_task

# Function to calculate expanding mean
def exp_mean(list_cols, list_train):
    target = list_cols[-1]
    df = pd.concat(list_train, axis=1)
    cumsum = df.groupby(list_cols[:-1])[target].cumsum() - df[target]
    cumcnt = df.groupby(list_cols[:-1]).cumcount()
    return(cumsum/cumcnt)

# Function to calculate mean encoding on a given dataset and return a dictionary with the encodings
# If there is a list of columns, the dictionary will have a tuple as its key 
def cal_mean_enc(list_cols, list_train):
    target = list_cols[-1]
    df = pd.concat(list_train, axis=1)
    mean_enc = df.groupby(list_cols[:-1]).agg(mean_enc=(target, 'mean')).to_dict()['mean_enc']
    return(mean_enc)

# Function to get key for the list of cols to which the mean encoding is applied. This key will be used to retrieve its 
# mean encoding 
def get_key(list_cols, df):
    first_col = list_cols[0]
    second_col = list_cols[1]
    return(list(zip(df[first_col], df[second_col])))

# This function combines the above 3 functions
def mean_encode(list_train, X_valid, features_to_engineer):
    for key, value in features_to_engineer.items():
        mean_enc = cal_mean_enc(value, list_train)
        X_train[key] = exp_mean(value, list_train)
        if len(value) > 2:
            X_valid[key] = get_key(value, X_valid)
            X_valid[key] = X_valid[key].map(mean_enc)
        else:
            X_valid[key] = X_valid[value[0]].map(mean_enc) 
        print(key,'created!')

Model Building¶

Since we need to predict expected time of arrival, I have used Mean Squared Error (MSE) as the loss function for all 3 machine learning models. There are tasks for which the time_taken is too high and it is possible that the time_taken as calculated for these tasks are not outliers. MSE is easier to optimize. Mean Absolute Error favors smaller values

Since we have about 6 categorical variables that have high cardinality, I have used 2 tree based models since they work much better than linear models/neural networks for data sets with a lot of categorical variables. Tree based models don't require you to one-hot encode the categorical variables which is necessary for linear models/neural networks. The 3 models that I have built are:

  • LightGBM (Gradient Boosting) with all features
  • Random Forest with only encoded features and continous variables (i.e no categorical variables)
  • A 4 layer Neural Network trained on scaled encoded features and continous variables (i.e no categorical variables)

These 3 models are diverse (boosting, bagging and neural networks) and depending on the features they have been trained on, they learn differently. In order to reduce the variance in estimates, I have taken a simple weighted average of the 3 models (weights based on RMSE on the validation set). Sufficient amount of hyperparameter tuning has been done through random search

End Notes¶

  • The time_taken as predicted on the validation set has an RMSE of 607 seconds, which is about 10 minutes. When you compare this to its standard deviation on the training set which is around 788 seconds, we have been able to explain part of the variability
  • The variability could have been reduced further if we had other important variables such as:
    • ID of the delivery partner: Are some delivery partners more efficient than others? Do some delivery partners know an area better than some other less-experienced delivery partners? If a delivery partner is selected for a particular task, can we predict with lesser variance as to what the time_taken would be based on his previous time_taken?
    • More temporal attributes: We only have data at a minute and seconds level. The assumption that we make when we predict time_taken at this level is that the time_taken is fairly constant for all hours of the day and all days of the week. Intuitively these attributes should matter
    • Location attributes: Although we have been given drop_area and pickupareaid, we don't have data about neighbouring locations. Thus an assumption that we make here is that the time taken is constant between any drop_area and pickupareaid that is D distance apart, regardless of which drop_area and pickupareaid we are talking about. Depending on neighboring locations and the traffic congestion in these places, the time taken could vary. Combined with other temporal attributes, it could even be powerful.
    • Is the task batched or not?: Are tasks batched? That is, does a delivery partner accept a task before completing one or more pending tasks? This would matter a lot as it is possible that he accepts the task quickly but he still has to complete other tasks before he starts on the new one
